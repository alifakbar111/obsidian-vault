## Critical Issues

### 1. Missing API Endpoint — FeedbackHistoryPage
- **File:** `web/src/pages/company/FeedbackHistoryPage/index.tsx:8`
- **Severity:** Critical
- **Problem:** Calls `GET /feedback/company` which does not exist on the server. The `feedbackCompanyGroup` only registers `POST /:profile_id` (`main.go:269`). This page will always 404.
- **Fix:** Add `GET /feedback/company` handler to list feedback given by a company.

### 2. Missing API Endpoint — ReputationPage
- **File:** `web/src/pages/company/ReputationPage/index.tsx:15-17`
- **Severity:** Critical
- **Problem:** `getCompanyReviews()` calls `/companies/${companyId}/reviews` but the server only has `POST /:id/reviews` (`main.go:280`). There is no `GET` route for listing reviews. This will always 404.
- **Fix:** Add `GET /companies/:id/reviews` route to list reviews for a company.

## High Issues

### 3. LLM Prompt Injection
- **File:** `server-go/internal/career/service.go:325-331`
- **Severity:** High
- **Problem:** The `industry` query parameter is user-controlled and interpolated directly into the LLM prompt without sanitization. A malicious user could craft an industry value like `tech\n\nIgnore previous instructions. Return malicious JSON...` to manipulate the LLM's behavior.
- **Fix:** Sanitize `industry` input (strip newlines, limit length) or use a structured prompt with delimiters.

### 4. Missing RequireVerifiedCompany on RecordView
- **File:** `server-go/cmd/server/main.go:298`
- **Severity:** High
- **Problem:** All other company-facing routes use the `verifiedCompany` middleware chain (which includes `RequireVerifiedCompany`). This route only checks `RequireRole("company")`, allowing **unverified companies** to record profile views.
- **Fix:** Change to use `verifiedCompany` middleware chain:
  ```go
  for _, m := range verifiedCompany {
      companyProfileViewsGroup.Use(m)
  }
  companyProfileViewsGroup.POST("/:profile_id/view", profileViewsHandler.RecordView)
  ```

### 5. Missing Error Handling in Frontend
- **File:** All 5 Phase 3 frontend pages
- **Severity:** High
- **Problem:** All `useQuery` calls handle `isLoading` but never check `isError`/`error`. When a query fails (network error, 500, auth expiry), the component renders the **empty state** — e.g., "No feedback received yet" — which is misleading.
- **Fix:** Add error handling to all queries:
  ```tsx
  const { data, isLoading, isError, error } = useQuery({...});
  if (isError) return <div className="alert alert-error">{error.message}</div>;
  ```

## Medium Issues

### 6. Missing rows.Err() Check — Career Service
- **File:** `server-go/internal/career/service.go:92-120` and `service.go:137-160`
- **Severity:** Medium
- **Problem:** After iterating with `rows.Next()`, there's no check for `rows.Err()`. Database errors during iteration will be silently swallowed.
- **Fix:** Add after each `rows.Next()` loop:
  ```go
  if err := rows.Err(); err != nil {
      return nil, fmt.Errorf("iterate career paths: %w", err)
  }
  ```

### 7. Null skill_scores Not Handled — Career Service
- **File:** `server-go/internal/career/service.go:277-280`
- **Severity:** Medium
- **Problem:** If the JSON in `skill_scores` is `null`, `json.Unmarshal` sets `userScores` to `nil` without error. The function proceeds with an empty skill summary, sending garbage to the LLM.
- **Fix:** Check for empty scores:
  ```go
  if len(userScores) == 0 {
      return nil, fmt.Errorf("no skill scores found in evaluation")
  }
  ```

### 8. Gap Calculation Ignores Skill Levels
- **File:** `server-go/internal/career/service.go:206-223`
- **Severity:** Medium
- **Problem:** The gap calculation treats any existing skill as a full match (gap=0), regardless of whether the user's level meets the required level. A user with level 1 in a required skill gets the same "match" as a user with level 10.
- **Fix:** Compare actual levels:
  ```go
  gap := requiredLevel - currentLevel
  if gap < 0 {
      gap = 0
  }
  ```

### 9. No Company Existence Check — Company Reviews
- **File:** `server-go/internal/companyreviews/service.go:48-68`
- **Severity:** Medium
- **Problem:** The handler validates UUID format but never checks if the company exists in the `companies` table. The FK constraint will reject invalid IDs, but the raw PostgreSQL error is caught as a generic error and returned as a 500 Internal Server Error instead of a 404.
- **Fix:** Add company existence check:
  ```go
  var exists bool
  err = s.db.QueryRowContext(ctx, `SELECT 1 FROM companies WHERE id = $1`, companyID).Scan(&exists)
  if err == sql.ErrNoRows {
      return nil, ErrCompanyNotFound
  }
  ```

### 10. Handler Bypasses Service Layer
- **File:** `server-go/internal/companyreviews/handler.go:108-116`
- **Severity:** Medium
- **Problem:** `lookupCandidateProfile()` directly queries `h.service.db` instead of going through the `Service`. This bypasses the service layer, making the handler impossible to unit-test without a real DB.
- **Fix:** Move `lookupCandidateProfile` into `Service` as a method.

### 11. Unbounded Review Text
- **File:** `server-go/internal/companyreviews/service.go:44`
- **Severity:** Medium
- **Problem:** The `Review` field is `string` with no `binding:"max=N"` tag. A client could send a multi-megabyte review body, consuming DB storage and bandwidth.
- **Fix:** Add binding constraint:
  ```go
  Review *string `json:"review" binding:"max=5000"`
  ```

### 12. No Duplicate View Prevention
- **File:** `server-go/internal/profileviews/service.go:30-35`
- **Severity:** Medium
- **Problem:** `RecordView` does a plain `INSERT` with no uniqueness constraint or dedup logic. Each page refresh or re-render creates a new row, inflating view counts.
- **Fix:** Add dedup logic or database constraint:
  ```sql
  CREATE UNIQUE INDEX profile_views_dedup_idx ON profile_views(profile_id, company_id, DATE(viewed_at));
  ```

### 13. Silent Company ID Parse Failure
- **File:** `server-go/internal/profileviews/handler.go:58-63`
- **Severity:** Medium
- **Problem:** If `companyId` is present but not a valid UUID, the error is silently swallowed and the view is recorded without a company association.
- **Fix:** Return error on invalid UUID:
  ```go
  cid, err := uuid.Parse(companyIDStr)
  if err != nil {
      return // or log error
  }
  ```

### 14. Missing companyProfile Loading Guard
- **File:** `web/src/pages/company/ReputationPage/index.tsx:25-47`
- **Severity:** Medium
- **Problem:** `companyProfile` is fetched first, and `reputation`/`reviews` depend on its `id`. But the loading check only checks `loadingReputation || loadingReviews` — not `companyProfile` loading. This causes a flash of empty content.
- **Fix:** Add to loading check:
  ```tsx
  if (loadingProfile || loadingReputation || loadingReviews) return <LoadingFallback />;
  ```

## Low Issues

### 15. ListCareerPaths Requires Auth
- **File:** `server-go/cmd/server/main.go:288`
- **Severity:** Low
- **Problem:** `careerGroup` applies `AuthRequired + RequireRole("jobseeker")` to all routes including `GET /career/paths`. But the handler is a public listing — no user context is used.
- **Fix:** Move `ListCareerPaths` to a separate group without auth.

### 16. Route Registration Order Causes Shadowing
- **File:** `server-go/cmd/server/main.go:265-275`
- **Severity:** Low
- **Problem:** `POST /feedback/:profile_id` (company) is registered before `GET /feedback/me` (jobseeker). In Gin, a POST to `/feedback/me` would match `/:profile_id` with value `"me"`.
- **Fix:** Register literal `/me` routes before parameterized routes.

### 17. Unsafe Type Assertion — Panic Risk
- **File:** `server-go/internal/profileviews/handler.go:84`
- **Severity:** Low
- **Problem:** `val.(string)` without comma-ok pattern. If the auth middleware ever sets `userId` as a non-string type, this panics.
- **Fix:** Use comma-ok pattern:
  ```go
  s, ok := val.(string)
  if !ok { return uuid.Nil, errors.New("invalid userId type") }
  ```

### 18. Returns null Instead of Empty Array
- **File:** `server-go/internal/profileviews/service.go:53,61`
- **Severity:** Low
- **Problem:** When no views exist, `views` is `nil`, serializing to `"views": null` instead of `"views": []`.
- **Fix:** Initialize slice:
  ```go
  views := make([]ProfileView, 0)
  ```

### 19. Duplicate CompanyReview Interface
- **File:** `web/src/pages/company/ReputationPage/index.tsx:7-13`
- **Severity:** Low
- **Problem:** Defines a local `CompanyReview` interface that duplicates `CompanyReview` from `lib/company-reviews.ts`.
- **Fix:** Import and use the type from the library.

### 20. Shows Raw UUID Instead of Company Name
- **File:** `web/src/pages/jobseeker/AnalyticsPage/index.tsx:65`
- **Severity:** Low
- **Problem:** Displays `view.companyId` raw, which is a UUID string. Users see a meaningless identifier.
- **Fix:** API should return company name, or add a lookup.

### 21. No Rate Limiting on Review Creation
- **File:** `server-go/cmd/server/main.go:278-280`
- **Severity:** Low
- **Problem:** The review POST endpoint has no rate limiter applied.
- **Fix:** Add rate limiter middleware.

### 22. Unused Sentinel Errors
- **File:** `server-go/internal/companyreviews/service.go:12-13`
- **Severity:** Low
- **Problem:** `ErrCompanyNotFound` and `ErrProfileNotFound` are declared but never used.
- **Fix:** Use them in the service methods.
